概念

纯函数是这样一种函数，即相同的输入，永远会得到相同的输出，而且没有任何可观察的副作用。

例子：

比如 slice 和 splice，这两个函数的作用并无二致——但是注意，它们各自的方式却大不同，但不管怎么说作用还是一样的。我们说 slice 符合纯函数的定义是因为对相同的输入它保证能返回相同的输出。而 splice 却会嚼烂调用它的那个数组，然后再吐出来；这就会产生可观察到的副作用，即这个数组永久地改变了。

redux实现原理

Veröffentlicht am 2018-06-12

1. 为什么用redux

在react中，数据在组件中是单向流动的，数据从一个党项父组件流向子组件（通过props），所以两个非父子组件之间通信就相对麻烦，redux的出现就是为了解决state里面的数据问题。

2. redux设计理念

redux 是将整个应用状态存储到一个地方上称为store，里面保存着一个状态树 store tree 如下图：

组件可以派发（dispatch）行为（action），而不是直接通知其他组件，组件内部通过订阅 store中的状态state来刷新自己的view。

3.redux 三大原则

  1. 唯一数据源
  2. 保持只读状态
  3. 数据改变通过纯函数来执行

1唯一数据源

整个应用的state都被存储到一个状态树里面，并且这个状态树只存在唯一的store中

2保持只读状态

state是制度的，唯一改变state的方式是触发action，action就是一个用于描述发生时间的普通对象

3数据改变只能通过纯函数来执行

使用纯函数来执行修改，action是通过reducers函数改变state的，其实是替换掉原来的state，如果忘了将原本的state返回回去的话，很容易造成数据的丢失，甚至会导致程序出错。

4. redux 概念解析

1. store

  • store 就是一个保存数据的地方，你可以把它看成一个数据，整个应用只能有一个store
  • redux 提供createStore 这个函数，用来生成store

2. state

state 就是store里面存储的数据，store里面可以有多个state，redux规定一个state对应一个view，只要state相同，view就是一样的，反过来也一样，可以通过store.getState()获取

3.Action
state 的改变会导致view的变化，但是在redux中不能直接操作state也饿就是说不能使用 this.state来操作，用户只能接触到View。在redux中提供一个对象来告诉store需要更新state。action是其中一个对象，type属性是必须的，表示action的名称，其他可以自由设定。

上面的type 是一个action的名称，就是通过这个action的名称做对应的操作。
payload 这个是参数 我一般用这个名称来携带参数

4.store.dispatch()
store.dispatch() 是view发出action的唯一办法

store.dispatch接受一个action作为参数，将它发送给store通知store来改变state。

5. reducer

store 收到action以后，必须给出一个新的state，这样view才会发生变化。这种state的计算过程就叫reducer。
reducer是一个纯函数，接受action和当前的state作为参数，返回一个新的state。

注意： reducer必须是一个纯函数，也就是说函数返回结果必须由state 和action决定，而且不产生任何副作用也不能修改state和 action 对象。

  1. console.log(1 + ‘2’ + ‘2’); // ‘122’

再次翻了一下红宝石，发现加法有个运算规则：

  1. 如果一个操作数是字符串，则将另外一个操作数转换成字符串，然后将两个字符串拼接起来

之前一直以为是根据左操作数来判别类型转换，今天做了这道题才知道，原来是这样子，之后要注意

原因，加号 + 还可以是一元运算符，就是将数值变为正数 ，例如：

在对非数值应用一元加操作符时，该操作符会像Number（）转型函数一样对这个值进行转换。换句话说，布尔false 和 true 会转换为 0 和 1。字符串会被按照一组则属的规则进行解析，而对象是先调用他们的valueOf() 和 toString（）方法，再转换得到值。

跨域问题

Veröffentlicht am 2018-05-27

1. 同源策略

同源策略限制从一个源加载的文档或脚本 如何与另一个源的资源进行交互。
一个源的定义： 如果协议，端口（如果指定了一个）和主机对于两个页面是相同的，则两个页面具有相同的源。

下表给出了相对 http://store.company.com/dir/page.html 同源检测的示例:

URL 结果 原因
http://store.company.com/dir/inner/another.html 成功 同一域名
http://store.company.com/dir2/other.html 成功 统一域名下的不同文件
https://store.company.com/secure.html 失败 不同的协议（https）
http://store.company.com:81/dir/etc.html 失败 不同的端口（81）
http://news.company.com/dir/other.html 失败 不同主机（news）

为什么要有跨域呢？

了解完跨域之后，想必大家都会有这么一个思考，为什么要有跨域的限制，浏览器这么做是出于何种原因呢。其实仔细想一想就会明白，跨域限制主要是为了安全考虑。

AJAX 同源策略主要是用来防止 CSRF 攻击。如果没有 ajax 同源策略，相当危险，每当我们发起一次http请求都会带上请求地址对应的cookie，那么可以做一些攻击：

  1. 用户登录了自己的银行页面 mybank.com，并保存了cookie
  2. 用户访问恶意页面 evil.com。执行了页面中的恶意ajax代码。
  3. evil.com向mybank.com 发送一个AJAX 请求，请求默认把mybank.com 对应cookie也同时发送过去
  4. 银行页面从发送的cookie中提取用户表示，验证用户无误，response 返回请求数据
  5. 由于ajax 在后台执行，用户无法感知这一过程。

DOM同源策略也一样，如果iframe之间可以跨域访问，可以这样攻击：

做一个假网站，里面用iframe嵌套一个银行网站 http://mybank.com
把iframe宽高啥的调整到页面全部，这样用户进来除了域名，别的部分和银行的网站没有任何差别。
这时如果用户输入账号密码，我们的主网站可以跨域访问到http://mybank.com 的dom节点，就可以拿到用户的输入了，那么就完成了一次攻击。
所以说有了跨域跨域限制之后，我们才能更安全的上网了。

跨域的解决方式

> 跨域资源共享

CORS 这是一个W3C标准 全称是：‘跨资源共享’（Cross-origin resource sharing）
对于这个方式，阮一峰老师总结的文章特别好，希望深入了解的可以看一下http://www.ruanyifeng.com/blog/2016/04/cors.html。

这里我就简单的说一说大体流程。

对于客户端，我们还是正常使用xhr对象发送ajax请求。
唯一需要注意的是，我们需要设置我们的xhr属性withCredentials为true，不然的话，cookie是带不过去的哦，设置： xhr.withCredentials = true;
对于服务器端，需要在 response header中设置如下两个字段:
Access-Control-Allow-Origin: http://www.yourhost.com
Access-Control-Allow-Credentials:true
这样，我们就可以跨域请求接口了。

MarginTop 为何影响父元素的 MarginTop

Veröffentlicht am 2018-05-19

在做弹窗的时候发现一个有趣的问题：
子元素用margin-top 将父元素顶下来了 ，并没有起到理想的效果，子元素相对于父元素顶部 XXXXpx
如下图：

红色边框 父元素 css：

绿色边框 第一个子元素 css ：

父元素并没有设置margin-top
但是 却又margin-top的效果
上网找了一下资料发现 这个 居然是 ==CSS2.1盒模型规范==

CSS2.1的盒模型中规定的内容——Collapsing margins：

In this specification, the expression collapsing margins means that adjoining margins (no non-empty content, padding or border areas or clearance separate them) of two or more boxes (which may be next to one another or nested) combine to form a single margin. 所有毗邻的两个或更多盒元素的margin将会合并为一个margin共享之。毗邻的定义为：同级或者嵌套的盒元素，并且它们之间没有非空内容、Padding或Border分隔。

这就是原因了。“嵌套”的盒元素也算“毗邻”，也会 Collapsing Margins。

这个问题的避免方法很多，只要破坏它出现的条件就行。给 Outer Div 加上 padding/border，或者给 Outer Div / Inner Div 设置为 float/position:absolute(CSS2.1规定浮动元素和绝对定位元素不参与Margin折叠)。 　　更多信息，关于不同值的margin折叠后表现的计算方法等等，可自行查阅W3C的CSS2.1规范：Collapsing margins。
