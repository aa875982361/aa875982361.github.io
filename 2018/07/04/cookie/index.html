Http cookie 是在客户端存储会话信息的。

设置方式

通过服务端在任意HTTP请求发送 Set-Http 头做为相应的一部分，其中包含会话信息。例如：

通过这样的返回值，浏览器会存储这样的信息，name=value，之后会为每个请求都会添加上cookie HTTP 请求头信息：

一开始，这个是一个比较好用的身份验证的方式，在登录成功之后，设置一个有效的token信息，之后发送请求就不用再次登录，服务器就知道你是谁。
但是这样存在一个缺点就是对每个请求都会发送cookie数据，如果是我从另外一个网站去调用目标网站的请求，也会带上相应的cookie，容易别人CSRF攻击。

后来的一些解决方式是将身份验证信息token存储在ls 和ss 里面，在请求头上面加一个自定义请求头，在发送请求之前带上token，

限制

为了确保cookie不会被卵用，都会做出相应的限制

数量限制：

每隔域的cookie数量都是有限制的，不同的的浏览器限制有点不同，当超过单个域名限制之后设置cookie，浏览器会清楚以前设置的cookie。ie和Opera会使用LRU算法，将最近的最少使用的cookie删除，腾出空间，

浏览器 每域名数量限制 大小限制
IE6 - 最多20 4095B
IE7+ 最多50 4095B
FireFox 最多50 4097B
Opera 最多30 4096B
sofari 和 chrome 没有限制 没有限制

构成

key 说明
名称 一个唯一确定的cookie，不区分大小写，必须是经过url编码的
值 字符串值，必须被url编码
域 cookie 对于那个域是有效的，向该域发送请求时，会自动加上cookie值，可以包含子域，默认是设置cookie的那个域
路径 对于指定域中的那个路径才向服务器发送cookie
失效时间 表示cookie身上时候被删除，一般来说，没有设置失效时间，默认是当前会话结束之后，就会把cookie删除掉，这个值必须是GMT时间格式（Wdy，DD-Mon-YYYY HH:MM:SS GMT）
安全标志 指定后，只有HTTPS才会发送相应的请求

这里要注意一下：域、路径、失效时间和secure标志都是服务器给浏览器的知识，以指定什么时候才发送cookie，这些参数并不会发送给服务端
