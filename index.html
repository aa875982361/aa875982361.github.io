2.Node vs Elemet

以下摘自MDN:

A Node is an interface from which a number of DOM types inherit, and allows these various types to be treated (or tested) similarly.
The following interfaces all inherit from Node its methods and properties: Document, Element, CharacterData (which Text, Comment, and CDATASection inherit), ProcessingInstruction, DocumentFragment, DocumentType, Notation, Entity, EntityReference.

HTMLCollection 和 NodeList的区别

Veröffentlicht am 2018-07-04

先介绍一下这两个概念：HTMLCollection 和 NodeList

  • HTMLCollection: 一个包含了元素（元素顺序为文档流中的顺序）的通用集合，还提供了用来从该集合中选择元素的方法和属性，
  • NodeList：是一个节点的集合，node.childNodes 返回，querySelectorAll 返回的

差异在于：

  • 方法： HTMLCollection 比NodeList 多一个节点
  • 包含方法节点类型不同 ： NodeList 可以包含任何节点类型，HTMLCollection 只包含袁术节点（ElementNode）

两个的出现场景：

  • 当返回多个几点（如：getElementByTagName) 或者得到所有子元素（如：element.childNodes)是，Collections就会出现，有可能返回 HTMLCollection 或 NodeList

看了这么多资料，感觉还是有点懵逼，还是喜欢从代码去看

HTMLCollection

HTMLCollection是以节点为元素的列表，可以凭借索引、节点名称、节点属性来对独立的节点进行访问。HTML DOM中的Collections是实时变动的，当原始文件变化，Collections也会随之发生变化。

  • 属性： length（返回的是列表的长度）
  • 方法一： item（通过序号索引获取节点，参数是索引值，超过索引值返回null）
  • 方法二：nameItem （用名字来返回一个节点，首先搜寻是否有匹配的id属性，如果没有就寻找是否有匹配的name属性，如果都买有，返回null）

Element 和 Node

文章中参考的是MDN：Node是一个基础类型，document, element, text, comment, DocumentFragment等都继承于Node. 在这篇文章最开始的测试中NodeList结果中有非常多的text，其实element, text, comment都是Node的子类，可以将它们视为：elementNode, textNode以及commentNode.平时在DOM中最常用的Element对象，其本质就是elementNode.
由于Node就是DOM的结构，代码内容经过解析后，Node与Node之间可以插入文本，文章最开头的截图中的”↵ “本质上就是Node之间的空隙，这种空隙的本质是textNode.

总结

HTMLCollection 和 NodeList 的共同点显而易见：

  1. 都是类数组对象，都有length
  2. 都有共同的方法： item，可以通过 item（index）或 item（id） 来访问元素
  3. 都是实时变动的，（例外： document.querySelectorAll 返回的NodeList 不是实时的

区别：

  1. NodeList 可以包含任何NODE节点类型,HTMLCollection 只包含元素节点（element_node) 标签
  2. HTMLCollection 比 NodeList多一项方法：nameditem 可以通过传递id或name属性来获得节点信息

一开始，这个是一个比较好用的身份验证的方式，在登录成功之后，设置一个有效的token信息，之后发送请求就不用再次登录，服务器就知道你是谁。
但是这样存在一个缺点就是对每个请求都会发送cookie数据，如果是我从另外一个网站去调用目标网站的请求，也会带上相应的cookie，容易别人CSRF攻击。

后来的一些解决方式是将身份验证信息token存储在ls 和ss 里面，在请求头上面加一个自定义请求头，在发送请求之前带上token，

限制

为了确保cookie不会被卵用，都会做出相应的限制

数量限制：

每隔域的cookie数量都是有限制的，不同的的浏览器限制有点不同，当超过单个域名限制之后设置cookie，浏览器会清楚以前设置的cookie。ie和Opera会使用LRU算法，将最近的最少使用的cookie删除，腾出空间，

FireFox 最多50 4097B
Opera 最多30 4096B
sofari 和 chrome 没有限制 没有限制

构成

key 说明
名称 一个唯一确定的cookie，不区分大小写，必须是经过url编码的
值 字符串值，必须被url编码
域 cookie 对于那个域是有效的，向该域发送请求时，会自动加上cookie值，可以包含子域，默认是设置cookie的那个域
路径 对于指定域中的那个路径才向服务器发送cookie
失效时间 表示cookie身上时候被删除，一般来说，没有设置失效时间，默认是当前会话结束之后，就会把cookie删除掉，这个值必须是GMT时间格式（Wdy，DD-Mon-YYYY HH:MM:SS GMT）
安全标志 指定后，只有HTTPS才会发送相应的请求

这里要注意一下：域、路径、失效时间和secure标志都是服务器给浏览器的知识，以指定什么时候才发送cookie，这些参数并不会发送给服务端

括号匹配方案

Veröffentlicht am 2018-06-30

合法的括号匹配序列被定义为:

  1. 空串””是合法的括号序列
  2. 如果”X”和”Y”是合法的序列,那么”XY”也是一个合法的括号序列
  3. 如果”X”是一个合法的序列,那么”(X)”也是一个合法的括号序列
  4. 每个合法的括号序列都可以由上面的规则生成
    例如””, “()”, “()()()”, “(()())”, “(((())))”都是合法的。 东东现在有一个合法的括号序列s,一次移除操作分为两步:
  5. 移除序列s中第一个左括号
  6. 移除序列s中任意一个右括号.保证操作之后s还是一个合法的括号序列
    东东现在想知道使用上述的移除操作有多少种方案可以把序列s变为空
    如果两个方案中有一次移除操作移除的是不同的右括号就认为是不同的方案。
    例如: s = “()()()()()”,输出1, 因为每次都只能选择被移除的左括号所相邻的右括号.
    s = “(((())))”,输出24, 第一次有4种情况, 第二次有3种情况, … ,依次类推, 4 3 2 * 1 = 24

输入描述:

输入包括一行,一个合法的括号序列s,序列长度length(2 ≤ length ≤ 20).

输出描述:

输出一个整数,表示方案数

输入例子1:

(((())))

输出例子1:

24

  2. 函数出口：空字符串就返回1
    例如 （（（））（））：

1. 页面被嵌入 iframe 中，重定向 iframe

先来说说我们的页面被嵌入了 iframe 的情况。也就是，网络运营商为了尽可能地减少植入广告对原有网站页面的影响，通常会通过把原有网站页面放置到一个和原页面相同大小的 iframe 里面去，那么就可以通过这个 iframe 来隔离广告代码对原有页面的影响。

window.self

window.top

返回窗口体系中的最顶层窗口的引用。

对于非同源的域名，iframe 子页面无法通过 ==parent.location== 或者 ==top.location== 拿到具体的页面地址，但是可以写入 ==top.location== ，也就是可以控制父页面的跳转。
